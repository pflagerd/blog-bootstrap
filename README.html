What is blog-bootstrap?

Purpose, philosophy, and a 30-second overview.

blog-bootstrap is a minimal personal blogging system designed to run directly from the filesystem — no web server required for day-to-day authoring and reading. It is intentionally built around Chromium (and Chromium-based browsers) as the sole runtime environment, which means every feature it uses is guaranteed to work in a consistent, modern browser context without polyfills or build steps.

It is meant to be either .forked (read dot-forked) or branched into another project. See the articles below for guidance on each approach.

Core philosophy: A blog should be as simple as a folder of files. Open index.html in Chromium and start writing. No npm, no build pipeline, no CMS login.

When you do want a live-reload authoring experience, a small Node.js + Puppeteer + Express server (index.js) can be launched to serve the files locally and keep a Chromium window open, saving window geometry between sessions.

.forking and Branching blog-bootstrap

Two ways to incorporate blog-bootstrap into another project.

.forking

If blog-bootstrap is .forked, its contents are copied to another repository, along with a .fork file containing the version of both repositories at the time of the dot-fork. You also add an entry in blog-bootstrap/.forks with the same information in a slightly different format.

You may consult this gist for guidance on how to a) create a .fork file, and b) how to update blog-bootstrap/.forks. There are prototype .fork and .forks files described there. Those prototypes contain instructions on how to put entries into them.

Branching

If it is branched, blog-bootstrap may be renamed because it will become a subdirectory of the containing project. The blog-bootstrap repository should be branched with the name of the containing project. If, for example, the containing project is called investing, the branch should be called investing also. The blog might be cloned at investing/blog. Since it is a nested repository, blog (in this case) should be added to investing/.gitignore.

Features

What blog-bootstrap does out of the box.

UI Collapsible Articles

Each <article> on the page gets an injected Expand / Collapse toggle automatically. The first article loads expanded; all others start collapsed. Collapse state is persisted per-article in localStorage, keyed by the article's id attribute (or an auto-generated fallback), so the page remembers your reading position across reloads.

Math KaTeX Rendering

All pages include the KaTeX library for inline and display math rendering. Use standard LaTeX delimiters: $…$ for inline, $$…$$ or \[…\] for display-mode equations.

Dev Optional Live-Reload Server

Running ./RUNME starts a local Express server on port 3000 and opens index.html in a non-headless Chromium window via Puppeteer. The window's width and height are saved to window-size.json on every resize (debounced to 500ms) and restored on next launch.

Copy Post Template

A hidden template article (style="display:none;") sits at the top of index.html. Copy it, remove the inline display style, and fill in your content.

How to Add a Post

Step-by-step guide to creating a new blog entry.

  1. Open index.html in your editor (or in Chromium DevTools with a Workspace mapping).
  2. Locate the template <article> block near the top of <section class="posts">.
  3. Copy the entire <article>…</article> block and paste it below the existing articles.
  4. Remove the style="display:none;" attribute from your new article.
  5. Give the article a unique id in YYMMDD-HHMM format so its expand/collapse state is tracked independently.
  6. Fill in the <h3> title, .meta summary line, date line, and body content.
  7. Reload in Chromium (location.reload() in the console, or Ctrl+R).
Note on localStorage and file://: Chromium permits localStorage on file:// URLs, but the storage is scoped to the exact file path. If you move the folder, expand/collapse memory resets. This is expected behavior.

inline-index-html.mjs

Generate a self-contained standalone version of any blog page.

This program can be used to generate a standalone version of index.html or meta-blog.html — a single file with all dependencies inlined. It is run thus: 

node inline-index-html.mjs input.html output.html

Where input.html is either index.html or meta-blog.html, and output.html is the name of the new standalone version.

Standalone versions are typically copied into another repository directly and are not dot-forked. The names inline-index.html and inline-meta-blog.html are commonly substituted for output.html, so they are listed in blog-bootstrap/.gitignore.

The Meta-Blog

A development journal that lives alongside the blog itself.

meta-blog.html is a companion page with the same CSS and JS as the main blog. Its purpose is to capture process writing — notes about what was built, why certain decisions were made, what's planned next, and any tooling or workflow observations.

It shares blog.css and blog.js, so any styling or behavior changes propagate to both pages automatically.

Authoring Directly in Chromium DevTools

No build step means DevTools Workspaces just work.

Because blog-bootstrap has no build step, Chromium's built-in Sources → Workspace feature works well for authoring. Map the project folder as a workspace, and edits made in DevTools save directly to disk. Combine with location.reload() in the console (or a keyboard shortcut macro) for a tight edit → reload loop without needing the Node server.

The meta-blog contains running notes on DevTools workflow improvements and wishlist items — including the desire for an automatic reload-on-save shortcut.

Running the Dev Server (Optional)

RUNME and SETMEUP handle prerequisites and startup automatically.

The recommended way to start the dev server is with the provided shell scripts, which handle prerequisites and startup in the right order: 

chmod +x RUNME   # once, if needed
./RUNME

SETMEUP — Prerequisite checker & installer

Sourced automatically by RUNME. Verifies that node and npm are on your $PATH, aborting with a clear message if either is missing. If both are present it runs npm ci to install exact dependency versions from package-lock.json — reproducible and safe for repeated runs.

RUNME — Entry-point launcher

Uses pushd/popd to cd to its own directory first, so it works correctly regardless of where you invoke it from. Sources SETMEUP, then delegates to node index.js, forwarding any command-line arguments via "$@".

Once running, a Chromium window opens at http://localhost:3000/index.html. Closing it shuts down the Express server automatically via the browser.on('disconnected') event in index.js.

The dev server is entirely optional. For reading and lightweight writing, simply open index.html directly in Chromium from the filesystem — no shell scripts needed.
